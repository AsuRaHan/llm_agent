# План переписывания кода агента на принципы SOLID, DRY, KISS

## Введение

Данный документ описывает план переписывания кода агента на основе принципов объектно-ориентированного программирования:

- **SOLID** — 5 принципов проектирования (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion)
- **DRY** — Don't Repeat Yourself (избегать повторений)
- **KISS** — Keep It Simple, Stupid (минимизировать сложность)
- **YAGNI** — You Aren't Gonna Need It (не реализовывать функции, которые пока не нужны)
- **SRP** — Single Responsibility Principle (принцип единственной ответственности)

## Детальный план по модулям

### Модуль 1: Core (Ядро)

#### `AssistantRole`
**Текущие проблемы:**
- Высокая ответственность (обработка запросов, планирование, управление инструментами)
- Зависимость от `ToolManager` и `LLMProvider`
- Сложная логика в `processQuery`

**План рефакторинга:**
1. **SRP (Single Responsibility Principle)**
   - Выделить `PlanningService` — сервис планирования задач
   - Выделить `ToolExecutionService` — сервис выполнения инструментов
   - Выделить `ResponseBuilder` — сервис построения ответов

2. **DRI (Dependency Inversion Principle)**
   - Зависимость от абстракций (`LLMProvider`, `ToolManager`)
   - Зависимость от `Config` — вынести в отдельный сервис

3. **DRY (Don't Repeat Yourself)**
   - Вынести повторяющуюся логику в сервисы

4. **KISS (Keep It Simple, Stupid)**
   - Упростить логику обработки запросов
   - Избегать сложных состояний

**Новая структура:**
```
src/
├── AssistantRole.h/cpp          # Координатор (только оркестрация)
├── PlanningService.h/cpp        # Планирование задач
├── ToolExecutionService.h/cpp   # Выполнение инструментов
├── ResponseBuilder.h/cpp        # Построение ответов
├── LLMService.h/cpp             # Обобщенный сервис LLM
└── ConfigService.h/cpp          # Сервис конфигурации
```

## Приоритеты рефакторинга

### Высокий приоритет (критические проблемы)
1. **AssistantRole** — главный класс с высокой ответственностью
2. **ToolManager** — управление инструментами
3. **ContextIndexer** — индексация контекста
4. **ApiHandlers** — обработка API

### Средний приоритет (важные проблемы)
1. **QueryProcessor** — обработка запросов
2. **MessageBuilder** — построение сообщений
3. **Config** — конфигурация
4. **Logger** — логирование

### Низкий приоритет (оптимизация)
1. **Инструменты** — рефакторинг по мере необходимости
2. **SessionManager** — управление сессиями
3. **WebSocketServer** — сервер WebSocket

## Критерии успешного рефакторинга

1. **Уменьшение циклометрии** — снижение сложности кода
2. **Уменьшение когнитивной нагрузки** — упрощение понимания кода
3. **Увеличение тестируемости** — возможность тестирования отдельных компонентов
4. **Улучшение читаемости** — код становится понятнее
5. **Снижение зависимости** — компоненты независимее друг от друга
6. **Уменьшение дублирования** — устранение повторяющегося кода
