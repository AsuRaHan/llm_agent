# План реализации поддержки инструментов (Tool Use) для агента «Умный молот» (обновлено 13.05.2026)

Этот документ описывает примерную стратегию и шаги по расширению функциональности AI-агента, наделяя его способностью использовать внешние инструменты (C++ функции) для выполнения действий и получения дополнительной информации.

## 1. Цель

Превратить агента из пассивного RAG-ответчика в проактивного помощника, который может:
- Взаимодействовать с файловой системой (читать, записывать, просматривать директории).
- Получать актуальную информацию, не вошедшую в первоначальный индекс (например, статус Git).
- Выполнять команды (например, запуск сборки или тестов).
- Задавать уточняющие вопросы и выполнять многошаговые задачи.

## 2. Ключевая концепция: Function Calling / Tool Use

Механизм работы будет основан на цикле взаимодействия с LLM:

1.  **Запрос пользователя:** Пользователь отправляет запрос (например, "Какие изменения есть в файле `main.cpp`?").
2.  **Описание инструментов:** Наше приложение передает в LLM не только запрос и RAG-контекст, но и JSON-описание доступных C++ функций (инструментов), таких как `read_file`, `git_status` и т.д.
3.  **Решение LLM:** Модель анализирует запрос и понимает, что для ответа ей нужно вызвать инструмент. Вместо генерации текста она возвращает специальный JSON-объект с намерением вызвать инструмент, например: `{"tool_calls": [{"name": "git_status", "arguments": {}}]}`.
4.  **Исполнение в C++:** Наше приложение парсит этот JSON, находит нужный инструмент (`git_status`) и выполняет соответствующую C++ функцию.
5.  **Передача результата:** Результат выполнения функции (например, вывод команды `git status`) снова отправляется в LLM как новое сообщение в диалоге с ролью `tool`.
6.  **Финальный ответ:** LLM, получив результат работы инструмента, генерирует итоговый, осмысленный ответ для пользователя (например, "В файле `main.cpp` были изменены следующие строки...").

## 3. Архитектурные изменения

### 3.1. `ToolManager`

Центральный компонент новой архитектуры.

- **`Tool` (структура):**
  - `std::string name`: Имя инструмента (e.g., `read_file`).
  - `std::string description`: Описание для LLM, что делает инструмент и какие у него параметры.
  - `nlohmann::json parameters`: JSON-схема, описывающая параметры функции (в формате OpenAI).
  - `std::function<std::string(const nlohmann::json&)> executor`: Лямбда-функция или указатель на функцию, которая будет выполнена. Принимает аргументы в виде JSON и возвращает результат строкой.

- **`ToolManager` (класс):**
  - `std::map<std::string, Tool> tools`: Реестр всех зарегистрированных инструментов.
  - `void registerTool(Tool tool)`: Метод для добавления нового инструмента.
  - `nlohmann::json getToolsSpecification() const`: Генерирует JSON-массив спецификаций всех инструментов для отправки в LLM.
  - `std::string executeTool(const std::string& name, const nlohmann::json& args)`: Находит инструмент по имени и вызывает его `executor` с переданными аргументами.

### 3.2. `AssistantRole`

Класс `AssistantRole` потребует значительной переработки.

- **`answerWithContext` -> `processQuery`:** Метод должен быть переделан для поддержки многошагового диалога с LLM.
- **Основной цикл в `processQuery`:**
  1. Инициализировать историю диалога (системный промпт, RAG-контекст, запрос пользователя).
  2. Начать цикл (с ограничением на количество итераций, например, 5, для предотвращения зацикливания).
  3. Отправить запрос в LLM с историей диалога и спецификацией инструментов из `ToolManager`.
  4. Проанализировать ответ:
     - Если это обычный текстовый ответ (`content`), вернуть его пользователю и завершить цикл.
     - Если это вызов инструмента (`tool_calls`), перейти к шагу 5.
  5. Для каждого вызова в `tool_calls`:
     - Вызвать `ToolManager::executeTool`.
     - Добавить в историю диалога результат выполнения с ролью `tool`.
  6. Вернуться к шагу 3 (отправить обновленную историю в LLM).

### Этап 2: Реализация продвинутых инструментов чтения и поиска

Вместо базового сканирования директорий, OmniCoder требует специализированных поисковых утилит для точечной навигации по кодовой базе.

4.  **`grep_search(pattern, path_pattern)`**
    - **Задача:** Эмуляция работы утилиты grep для поиска конкретных строк, сигнатур, методов или регулярных выражений по файлам проекта.
    - **Реализация в C++:** Использовать `std::regex` при построчном чтении файлов. Ограничить область поиска через маску `path_pattern` (например, `*.cpp`, `*.h`).
    - **Польза:** Избавляет от необходимости скармливать модели файлы целиком, если нужно найти только место вызова конкретной функции.

5.  **`file_glob_search(pattern)`**
    - **Задача:** Поиск файлов по шаблону имени (глоббинг).
    - **Реализация в C++:** Использовать `std::filesystem` для фильтрации файлов в проекте по маске (например, `*Indexer*`).
    - **Польза:** Позволяет модели мгновенно находить нужные заголовочные файлы или конфиги, зная лишь часть их имени.

### Этап 3: Реализация инструментов модификации кода (Уровень Smart-Агента)

OmniCoder эффективен тогда, когда он правит код не «перезаписью всего файла», а точечными патчами. Это минимизирует поломку синтаксиса и экономит контекст.

1.  **`edit_file(path, target_line, old_text, new_text)`**
    - **Задача:** Безопасная замена конкретного блока кода внутри файла.
    - **Безопасность:** **Требует подтверждения пользователя.** C++ код должен найти строку `target_line`, проверить, совпадает ли оригинальный текст на ней с `old_text`, и только в случае успеха применить `new_text`.
    - **Польза:** Самый надежный способ точечного рефакторинга без риска стереть половину файла из-за сбоя генерации LLM.

2.  **`apply_diff(path, diff_content)`**
    - **Задача:** Применение стандартного унифицированного патча (Unified Diff / Git Diff) к указанному файлу.
    - **Безопасность:** **Требует подтверждения пользователя.** Реализуется через парсинг блоков `@@ -l,s +l,s @@` или вызов системной утилиты `patch`.
    - **Польза:** Позволяет модели за один вызов внести комплексные связанные изменения в несколько мест одного файла (например, добавить объявление метода в класс и его реализацию ниже).

3.  **`get_datetime()`**
    - **Задача:** Возврат текущей даты, времени и временной зоны системы.
    - **Реализация в C++:** Стандартный `std::chrono::system_clock`.
    - **Польза:** Необходим модели для корректной генерации логов, временных меток в файлах или анализа актуальности кэша сборки.
