# CHANGELOG

## [0.1.6] - 2026-05-22

### 🚀 Основные улучшения

#### 1. **Рефакторинг движка агента (QueryProcessor)**
- **Упрощение логики обработки запросов**: Удалена сложная двухфазная архитектура (Thinking Phase + ReAct Loop) в пользу единого цикла ReAct
- **Улучшенная обработка ответов LLM**: 
  - Добавлена четкая обработка 3 сценариев: прямой ответ, вызов инструментов, пустой ответ
  - Автоматическое подведение итогов при получении пустого ответа от модели
- **Новый метод `getSummaryAnswer()`**: Генерирует итоговый ответ на основе всей истории диалога

#### 2. **Упрощение состояния сессий**
- **Удалены избыточные статусы**:
  - `GENERATING_PLAN` → объединено с `THINKING`
  - `AWAITING_PLAN_CONFIRMATION` → удалено (планирование теперь встроенное)
  - `EXECUTING_PLAN` → удалено (единый цикл ReAct)
- **Удалены поля сессии**:
  - `plan_steps` — шаги плана теперь генерируются динамически
  - `current_plan_step` — не требуется
  - `original_user_query` — не требуется

#### 3. **Увеличение лимита вызовов инструментов**
- `max_tool_calls`: **50 → 150** (в `Config.h`)

#### 4. **Упрощение WebSocket-сервера**
- **Удалены обработчики**:
  - `handlePlanConfirmation()` — больше не требуется
  - `processPlanGeneration()` — встроен в основной цикл
- **Упрощена логика обработки запросов**:
  - Все запросы теперь проходят через единый цикл `processAgentLogic()`
  - Удалена лингвистическая эвристика для определения необходимости планирования
  - RAG-контекст собирается для каждого запроса динамически

#### 5. **Улучшение обработки ошибок**
- **Единая логика восстановления**:
  - Ошибки теперь обрабатываются одинаково как в плановом, так и в обычном режиме
  - Удалена фильтрация опции `re-plan` в одношаговом режиме
- **Улучшен сброс сессий**:
  - При очистке сессий сбрасываются только `history`, `status` и `pending_tool_call`

### ✅ Преимущества изменений

1. **Упрощение кода**: -498 строк, меньше сложной логики
2. **Улучшенная надежность**: Единый цикл ReAct, меньше точек отказа
3. **Гибкость**: Динамическое планирование вместо жесткой структуры
4. **Производительность**: Меньше состояния сессий, быстрее сериализация
5. **Поддержка**: Проще поддерживать и расширять

### ⚠️ Миграционные заметки

#### Для разработчиков:
- Убедитесь, что фронтенд не ожидает сообщения `confirm_plan`
- Удалите из фронтенда обработчики событий `plan_generated` и `plan_update`
- Обновите структуру сессий в базе данных (удалите лишние поля)

#### Для пользователей:
- Автономное планирование теперь работает "под капотом" без явного подтверждения
- Лимит вызовов инструментов увеличен до 150
