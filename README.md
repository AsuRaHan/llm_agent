# Project: Smart Hammer (Local AI Context Indexer)

## 🎯 Общая концепция
Создание "умного инструмента" для разработчика, который превращает локальный исходный код в динамическую базу знаний. В отличие от облачных решений, этот проект работает локально, обеспечивая полную приватность и минимальную задержку.

## 🧠 Зачем это нужно
LLM (Large Language Model) интеллект ограничен объемом оперативной памяти. Инструмент должен стать "экзокортексом" (внешним расширением мозга), который:
1. Помнит структуру и логику всего проекта.
2. Автоматически индексирует изменения.
3. Позволяет общаться с кодом на естественном языке через локальные LLM.

## 🛠 Технологический стек
- **Язык:** C++20 (использование современных фишек: std::filesystem, std::ranges).
- **Сборка:** CMake (FetchContent для управления зависимостями).
- **Зависимости:** 
  - `yhirose/cpp-httplib` (сетевое взаимодействие).
  - `nlohmann/json` (хранение данных и работа с API).
- **AI Backend:** Локальный сервер `llama.cpp` (OpenAI-compatible API) на `localhost:8080`.
- **Hardware:** Оптимизировано под NVIDIA RTX 5060 Ti (16GB VRAM).

## 🏗 Архитектура
1. **ContextIndexer:** Рекурсивно обходит файлы проекта, фильтрует мусор (build, .git) и отслеживает изменения через хеширование (std::hash).
2. **EmbeddingClient:** Отправляет контент файлов на локальный AI-сервер для получения "смысловых отпечатков" (эмбеддингов) или саммари.
3. **Storage:** Хранит индекс в формате JSON для прозрачности и легкого доступа.

## 📜 Инструкции для ИИ-Агента
- **Автономность:** Ты имеешь право самостоятельно изменять файлы и предлагать улучшения архитектуры.
- **Чистота кода:** При конфликтах с Windows.h всегда используй `#define WIN32_LEAN_AND_MEAN` и `#define NOMINMAX`.
- **Логирование:** Каждое значимое изменение должно отражаться в `ai_log.md`.
- **Приоритет:** Сначала стабильный билд через CMake, затем наращивание функционала.

# Руководство пользователя: Smart Hammer

Этот документ описывает, как настроить и использовать проект "Smart Hammer".

## 1. Обзор

"Smart Hammer" — это локальный ИИ-ассистент для разработчиков. Он индексирует исходный код вашего проекта, создавая базу знаний, с которой можно взаимодействовать на естественном языке. Это позволяет быстро находить релевантные фрагменты кода и получать их анализ от большой языковой модели (LLM).

Ключевая особенность — **полная локальность**. Все ваши данные и код остаются на вашей машине, обеспечивая приватность и высокую скорость работы.

## 2. Системные требования и настройка

### Обязательное требование: Локальный ИИ-сервер

Проект **не будет работать** без запущенного локального сервера, совместимого с OpenAI API. Рекомендуется использовать `llama.cpp`.

1.  **Скачайте и соберите `llama.cpp`**: Следуйте официальной инструкции на GitHub-странице llama.cpp.
2.  **Скачайте модель**: Вам понадобится модель в формате GGUF, поддерживающая создание эмбеддингов (смысловых векторов). Например, модели из серии `bge`.
3.  **Запустите сервер**: Запустите сервер `llama.cpp` в режиме совместимости с OpenAI API. Он должен быть доступен по адресу `http://localhost:8080`.

    Пример команды для запуска сервера:
    ```bash
    ./server -m your-model.gguf -c 4096 --host 0.0.0.0 --port 8080 --embedding
    ```
    - `-m`: путь к вашей модели.
    - `-c`: размер контекста.
    - `--embedding`: флаг, включающий поддержку эмбеддингов.

    **Важно:** Убедитесь, что сервер запущен и отвечает на запросы, прежде чем запускать "Smart Hammer".

### Сборка проекта

Проект использует CMake для сборки.

1.  **Клонируйте репозиторий**:
    ```bash
    git clone <URL-вашего-репозитория>
    cd llm_agent
    ```
2.  **Создайте директорию для сборки**:
    ```bash
    mkdir build
    cd build
    ```
3.  **Сконфигурируйте и соберите проект**:
    ```bash
    cmake ..
    cmake --build .
    ```
    В Windows можно использовать Visual Studio для открытия сгенерированного проекта.

## 3. Использование

После успешной сборки исполняемый файл (например, `llm_agent.exe` в Windows) будет находиться в директории `build/src/Debug` или `build/src/Release`.

1.  **Запуск**: Запустите исполняемый файл из корневой директории проекта (`llm_agent`).
    ```bash
    # Находясь в папке llm_agent/
    ./build/src/Debug/llm_agent
    ```

2.  **Индексация**: При первом запуске программа начнет индексировать все файлы в текущей директории и ее поддиректориях. Она пропустит системные папки (вроде `.git`, `build`) и бинарные файлы. Для каждого файла будет создан "смысловой отпечаток" (эмбеддинг) с помощью вашего локального ИИ-сервера. Процесс может занять некоторое время. Результаты сохраняются в файл `index.json`.

3.  **Поиск**: После индексации программа предложит вам ввести поисковый запрос.
    ```
    Введите поисковый запрос для нахождения наиболее похожего файла (или нажмите Enter для выхода):
    ```
    Введите вопрос о вашем коде. Например: "как работает обработка ошибок в API?" или "где реализована функция аутентификации?".

4.  **Анализ**: Программа найдет наиболее релевантный файл в вашем проекте и спросит, хотите ли вы его проанализировать.
    ```
    Наиболее похожий файл: src/ErrorHandler.cpp (схожесть: 0.85)
    Хочешь, чтобы я проанализировал этот файл? (y/n):
    ```
    - Введите `y` (или `Y`), чтобы отправить содержимое файла и ваш первоначальный вопрос на анализ в LLM.
    - Введите `n`, чтобы пропустить анализ и ввести новый поисковый запрос.

5.  **Результат**: Если вы согласились на анализ, вы получите ответ от ИИ, основанный на содержимом найденного файла и вашем вопросе.

6.  **Выход**: Чтобы завершить работу программы, просто нажмите `Enter` в ответ на приглашение ввести поисковый запрос.

## Что именно попадает в дефолтный `config.json`

Функция `create_default` (см. `Config.cpp`) формирует JSON следующего вида (пример, с отступами `4`):

```json
{
    "server": {
        "host": "localhost",
        "port": 8080,
        "retry_count": 3,
        "retry_delay_ms": 500
    },
    "embedding": {
        "max_text_length": 1500
    },
    "assistant": {
        "chat_completion_timeout_sec": 300
    },
    "indexing": {
        "chunk_size": 1000,
        "chunk_overlap": 200,
        "top_k_results": 3,
        "ignored_directories": [
            "build", ".git", ".vscode", "CMakeFiles"
        ],
        "ignored_extensions": [
            ".exe", ".obj", ".pdb", ".ilk", ".sln", ".vcxproj", ".filters", ".user",
            ".recipe", ".tlog", ".lastbuildstate", ".bin", ".stamp", ".cmake",
            ".json", ".log"
        ],
        "ignored_files": [ ".gitignore" ]
    }
}
```

Эти значения берутся из полей‑по‑умолчанию в `Config.h` и из локальных констант в `Config.cpp`.

## Что происходит с остальными параметрами

- **Если файл успешно загружен** (после создания дефолта или если он уже существовал), все параметры берутся из JSON.  
- **Если в JSON есть ключ, но его значение отсутствует** (например, только `"server"` без `"host"`), используется значение‑по‑умолчанию из структуры `Config` (см. `j.value(..., default)`).  
- **Если загрузка полностью не удалась** (например, JSON повреждён), вызывается `catch (const json::exception& e)`, пишется `SPDLOG_CRITICAL` и функция возвращает `false`.
