# 🔨 Smart Hammer

**Smart Hammer** — это продвинутый локальный AI-ассистент для разработчиков, созданный для глубокого анализа, понимания и взаимодействия с кодовой базой ваших проектов. 

В отличие от традиционных инструментов поиска кода, Smart Hammer использует **семантическое понимание** и **векторные эмбеддинги** для ответа на сложные вопросы, поиска информации по смыслу, а не просто по ключевым словам, и даже выполнения задач по модификации кода.

---

## 🌟 Ключевые возможности

### 🧠 Семантический поиск и понимание кода
- **Глубокий анализ кода:** Использование `tree-sitter` для синтаксического разбора и выделения семантических блоков (функции, классы, методы, переменные)
- **Векторные эмбеддинги:** Преобразование кода в векторные представления для поиска по смыслу
- **Контекстный поиск:** Найдите не только "где эта функция", но и "как работает эта логика" или "где используются подобные паттерны"

### 💬 Интеллектуальный чат с кодом
- Задавайте вопросы о структуре проекта
- Спрашивайте о назначении конкретных функций и модулей
- Получайте объяснения сложных алгоритмов
- Находите связанные файлы и зависимости

### ✏️ Интерактивное редактирование кода
- Безопасное чтение, запись и редактирование файлов
- Применение сложных изменений через unified diff патчи
- Поддержка атомарных операций с проверкой изменений
- Автоматическое сохранение истории изменений

### 🔧 Расширяемая система инструментов
Smart Hammer предоставляет набор инструментов для взаимодействия с файловой системой и кодом:

| Инструмент | Описание |
|------------|----------|
| `read_file` | Чтение содержимого файлов |
| `write_file` | Запись/дописывание контента в файлы |
| `edit_file` | Безопасное точечное редактирование кода |
| `apply_diff` | Применение unified diff патчей |
| `code_search` | Семантический поиск по кодовой базе |
| `grep_search` | Регулярные выражения для поиска |
| `file_glob_search` | Поиск файлов по glob-паттернам |
| `list_directory` | Просмотр содержимого директорий |
| `execute_shell_command` | Выполнение команд терминала (с подтверждением) |
| `github_repository_search` | Поиск репозиториев на GitHub |
| `read_url` | Загрузка и чтение веб-страниц |
| `open_url_in_browser` | Открытие URL в браузере |
| `get_datetime` | Получение текущей даты и времени |
| `get_system_info` | Информация об операционной системе |

### 🌐 Веб-интерфейс
- Удобный и отзывчивый интерфейс на HTML/CSS/JavaScript
- Реальное время обновление чата
- Поддержка истории диалогов
- Доступен через любой современный браузер

### 🔄 Мониторинг изменений
- Автоматическое отслеживание изменений в файлах
- Динамическое обновление векторного индекса
- Поддержка hot-reload для разработки

```
│  │ (HTML/JS)    │    │  (C++20)     │    │  (External)  │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
│         │                    │                    │         │
│         │ WebSocket          │                    │         │
│         │                    │                    │         │
│         ▼                    ▼                    ▼         │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐  │
│  │  WebSocket   │    │  Context     │    │  OpenAI API  │  │
│  │  Client      │    │  Indexer     │    │  Provider    │  │
│  └──────────────┘    └──────────────┘    └──────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### Backend (C++20) — Ядро системы

**Основные компоненты:**

1. **ContextIndexer** — менеджер индексации кодовой базы
   - Рекурсивный обход директорий
   - Фильтрация по расширениям и именам
   - Параллельная обработка файлов

2. **CodeParser (tree-sitter)** — синтаксический анализатор
   - Выделение семантических блоков
   - Определение типов кода (функции, классы, переменные)
   - Извлечение метаданных

3. **EmbeddingGenerator** — генерация векторных представлений
   - Интеграция с LLM-сервером
   - Создание эмбеддингов для каждого блока кода
   - Хранение в векторном индексе

4. **HNSWIndex** — векторный индекс (hnswlib)
   - Эффективный поиск по сходству
   - Поддержка динамического добавления/удаления
   - Оптимизированная память

5. **AssistantRole (ReAct)** — логика принятия решений
   - Цикл Reason-Act для обработки запросов
   - Выбор инструментов на основе контекста
   - Генерация ответов и вызовов инструментов

6. **ToolManager** — диспетчер инструментов
   - Реализация всех доступных инструментов
   - Безопасность и валидация
   - Логирование операций

7. **WebServer (cpp-httplib)** — веб-сервер
   - WebSocket для реального времени
   - REST API для управления
   - Статический хостинг фронтенда

### Frontend — Пользовательский интерфейс

- **Чат-интерфейс:** История сообщений, ввод запросов
- **Статус-панель:** Прогресс индексации, активные операции
- **Настройки:** Конфигурация LLM-сервера, параметры поиска
- **Консоль вывода:** Логи системы в реальном времени

### LLM-сервер — Внешний компонент

Smart Hammer **не включает** LLM в себя. Он взаимодействует с внешним сервером, который должен предоставлять:
- `/v1/chat/completions` — для генерации ответов
- `/v1/embeddings` — для создания векторных представлений

**Рекомендуемые решения:**
- `llama.cpp` с флагом `--embedding`
- `Ollama`
- `Text Generation WebUI`
- Любой OpenAI-совместимый сервер

## 🚀 Запуск и использование

### Первая настройка

1. **Запустите ассистент** с указанием директории для индексации:
   ```bash
   ./Agent /путь/к/вашему/проекту
   ```

2. **Настройте конфигурацию:**
   - При первом запуске создастся `.shdata/config.json`
   - Отредактируйте секцию `"server"`:
   ```json
   {
     "server": {
       "host": "localhost",
       "port": 8080,
       "timeout": 30
     },
     "index": {
       "ignored_directories": [".git", "node_modules", "build"],
       "ignored_extensions": [".min.js", ".map"]
     }
   }
   ```

3. **Откройте веб-интерфейс:**
   - Следуйте ссылке в консоли (по умолчанию `http://localhost:9000`)
   - Начните задавать вопросы о вашем коде!

### Примеры использования

## 💡 Как это работает

### Процесс индексации

```
1. Сканирование директорий
   └─> Рекурсивный обход с фильтрацией

2. Парсинг файлов
   └─> tree-sitter → семантические блоки

3. Генерация эмбеддингов
   └─> LLM API → векторные представления

4. Сохранение индекса
   └─> HNSW → .shdata/index.bin
```

### Цикл обработки запроса (ReAct)

```
Пользовательский запрос
    ↓
Семантический поиск по индексу
    ↓
Сбор релевантных фрагментов
    ↓
Формирование промпта (вопрос + контекст + история)
    ↓
LLM анализирует и решает:
    ├─ Ответить напрямую
    └─ Вызвать инструмент
         ↓
    Выполнение инструмента
         ↓
    Добавление результата в историю
         ↓
    Повторение цикла (если нужно)
         ↓
    Финальный ответ
```

### Пример взаимодействия

```
Пользователь: "Где хранятся настройки приложения?"

1. Поиск: находит упоминания "config", "settings"
2. LLM решает: нужно прочитать config.json
3. Инструмент: read_file(.shdata/config.json)
4. LLM анализирует результат
5. Ответ: "Настройки хранятся в .shdata/config.json. Основные параметры: server.host, server.port..."
```

## 🔒 Безопасность

### Принципы безопасности

- **Подтверждение команд:** Все shell-команды требуют явного подтверждения
- **Изоляция:** Ассистент работает только в указанной директории
- **Валидация:** Все входные данные проверяются на безопасность
- **Логирование:** Все операции логируются для аудита

### Рекомендации

1. Не запускайте ассистента с доступом к чувствительным файлам
2. Регулярно обновляйте зависимости
3. Используйте `.gitignore` для игнорирования нежелательных файлов
4. Настройте `ignored_directories` в конфигурации
