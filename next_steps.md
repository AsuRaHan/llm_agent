# План реализации режима "Plan-and-Execute"

## I. Общая концепция

Превратить "Smart Hammer" из реактивного чат-бота в полноценного автономного инженера. Это достигается внедрением двухэтапного режима работы:

1.  **Планирование (Planning):** На сложный запрос пользователя ("Перепиши X", "Исправь баг Y") агент сначала не выполняет работу, а генерирует детальный JSON-план, состоящий из атомарных шагов. Этот план представляется пользователю на утверждение.
2.  **Выполнение (Execution):** После утверждения плана агент переходит в автономный режим и последовательно выполняет каждый шаг, используя инструменты, отчитываясь о прогрессе и не требуя вмешательства пользователя до полного завершения задачи или возникновения ошибки.

## II. Изменения в бэкенде (C++)

### 1. Модификация структур данных

*   **`UserSession.h`**: Расширить сессию для поддержки состояния планирования.
    *   Добавить новые статусы в `enum AgentStatus`:
        *   `GENERATING_PLAN`: Модель составляет план.
        *   `EXECUTING_PLAN`: Агент выполняет утвержденный план.
    *   Добавить поля для хранения плана:
        *   `nlohmann::json plan_steps;` // JSON-массив шагов
        *   `int current_plan_step = -1;` // Индекс текущего шага
        *   `std::string original_user_query;` // Исходный запрос для RAG

*   **`AssistantRole.h`**: Модифицировать `AssistantResponse`, чтобы он мог сигнализировать о незавершенности плана.
    *   Добавить флаг `bool plan_completed = true;`. Если `false`, значит агент выдал промежуточный текстовый ответ, но основной план еще выполняется.

### 2. Реализация логики планирования

*   **`AssistantRole.cpp`**: Создать новый метод `generatePlan`.
    *   Метод `generatePlan(const std::string& user_query)` будет делать скрытый запрос к LLM.
    *   Он будет использовать специальный системный промпт, заставляющий модель вернуть ответ **строго** в формате JSON-массива строк.
    *   Необходимо предусмотреть фолбэк-парсер на случай, если модель вернет JSON внутри markdown-блока.

### 3. Реализация логики выполнения

*   **`WebSocketServer.cpp`** (внутри `processAgentLogic` или аналогичного обработчика в пуле потоков):
    *   **Начало выполнения:** При получении от клиента `confirm_plan`, перевести сессию в статус `EXECUTING_PLAN`, установить `current_plan_step = 0` и запустить основной цикл выполнения.
    *   **Основной цикл:**
        1.  Взять текущий шаг из `session->plan_steps[session->current_plan_step]`.
        2.  Сформировать системное сообщение `[SYSTEM NOTE]: Отличная работа. Теперь переходи к шагу: "..."` и добавить его в историю диалога.
        3.  Вызвать `assistant.processQuery()`, передав ему обновленную историю. `queryText` при этом должен быть пустым, чтобы не вызывать повторный RAG-поиск.
        4.  Обработать ответ `AssistantResponse`.
        5.  Если `plan_completed` все еще `false` (агент просто отчитался текстом или использовал инструмент), инкрементировать `current_plan_step` и перейти к следующей итерации цикла.
        6.  Отправлять на фронтенд сообщение `plan_update` с текущим прогрессом.
    *   **Завершение:** Когда все шаги выполнены (`current_plan_step >= plan_steps.size()`), выйти из цикла, перевести сессию в `IDLE` и отправить финальный ответ.

### 4. Интеграция с WebSocket

*   **`WebSocketServer.cpp`**:
    *   Добавить обработку входящего сообщения `confirm_plan`.
    *   Реализовать отправку исходящих сообщений:
        *   `plan_generated`: после успешной генерации плана, содержит массив шагов.
        *   `plan_update`: на каждой итерации выполнения, содержит массив шагов и индекс текущего.

## III. Изменения во фронтенде (JavaScript)

### 1. Обработка новых сообщений от сервера

*   В `socket.onmessage` добавить `case` для `plan_generated` и `plan_update`.

### 2. Виджет подтверждения плана

*   Создать функцию `addPlanConfirmationWidget(steps)`.
*   Она должна отрендерить список шагов и две кнопки: "Утвердить и запустить" и "Отмена".
*   При нажатии на "Утвердить" отправлять на сервер сообщение `{ type: 'confirm_plan', data: { confirmed: true } }`.
*   На время ожидания утверждения блокировать поле ввода.

### 3. Виджет прогресса выполнения

*   Создать функцию `updatePlanProgress(currentStepIndex, steps)`.
*   Она должна рендерить динамический список задач с иконками статуса (✅ Выполнено, ⚙️ Выполняется, ⏳ Ожидает).
*   Виджет должен иметь `id="active-plan-progress"`, чтобы его можно было найти и обновить на следующей итерации.

### 4. Очистка

*   После получения финального ответа `query_response` необходимо убрать `id` у виджета прогресса (`oldProgress.removeAttribute('id')`), чтобы при следующем запуске плана создался новый виджет, а не перезаписался старый.
