# План на будущее: Асинхронность, Визуализация и Расширение возможностей

Привет! Это наш план по дальнейшему усовершенствованию "Умной Кувалды". Мы сосредоточимся на трех ключевых направлениях: производительность сервера, удобство интерфейса и новые возможности для анализа кода.

## Фаза 1: Асинхронный WebSocket-сервер (Повышение производительности)

**Проблема:** Сейчас долгий запрос к LLM (например, на сложный анализ кода) может "подвесить" весь WebSocket-сервер. Другие пользователи (или даже другие действия в той же вкладке) будут ждать, пока не завершится текущая операция.

**Цель:** Сделать сервер полностью асинхронным, чтобы он мог одновременно обрабатывать множество запросов без блокировок.

**План действий:**

1.  **Создание пула потоков:** В классе `ApiHandlers` или `WebSocketServer` мы создадим простой пул потоков (`std::vector<std::thread>`) и потокобезопасную очередь задач.
2.  **Рефакторинг обработчиков:** Методы `handleQuery` и `handleConfirmation` больше не будут выполнять всю работу сами. Вместо этого они будут "упаковывать" запрос (ID сессии, текст запроса, callback для ответа) в задачу и помещать ее в очередь.
3.  **Фоновое выполнение:** Свободный поток из пула заберет задачу, вызовет `assistant->processQuery`, дождется ответа от LLM и, используя сохраненный callback, отправит результат обратно клиенту.

**Результат:** Сервер станет отзывчивым и масштабируемым.

## Фаза 2: Визуализация изменений кода (Улучшение UI/UX)

**Проблема:** Когда агент предлагает изменить файл (`edit_file`, `apply_diff`), пользователь видит только текстовое описание. Это не очень наглядно и заставляет доверять агенту "вслепую".

**Цель:** Показывать пользователю наглядное сравнение "было/стало" (diff) прямо в интерфейсе чата перед подтверждением.

**План действий:**

1.  **Новый тип сообщения `diff_required`:** Когда агент предлагает изменить файл, он будет отправлять на фронтенд это специальное сообщение.
2.  **Передача данных для diff:** Сообщение будет содержать `old_content` (исходный код) и `new_content` (код после правок).
3.  **Рендеринг на фронтенде (JS):**
    *   При получении `diff_required`, мы будем использовать легковесную JS-библиотеку (например, `diff2html` или даже простую самописную) для генерации HTML-представления изменений.
    *   Этот HTML будет вставлен в окно чата, показывая пользователю подсвеченные красным и зеленым цветом строки.
    *   Кнопки "Применить" и "Отклонить" будут относиться именно к этому diff.

**Результат:** Пользователь будет принимать осознанные решения, видя точные изменения, предлагаемые агентом.

## Фаза 3: Загрузка и анализ отдельных файлов

**Проблема:** Сейчас агент может анализировать только те файлы, которые были в проекте на момент первоначальной индексации.

**Цель:** Дать пользователю возможность "на лету" загружать файлы (например, логи, сторонние скрипты, документы) для анализа.

**План действий:**

1.  **Новый HTTP-эндпоинт `/api/upload`:** В `ApiHandlers` мы создадим роут, который будет принимать файлы через `multipart/form-data`.
2.  **Обработка на бэкенде:** Сервер сохранит загруженный файл во временную директорию (например, `.shdata/uploads`). После этого он инициирует новый диалог с агентом, передавая ему системное сообщение: "Пользователь загрузил файл `file.txt`. Проанализируй его содержимое."
3.  **Кнопка на фронтенде (HTML/JS):**
    *   В `chat.html` добавим кнопку "Загрузить файл" (`<input type="file">`).
    *   JavaScript будет перехватывать выбор файла и отправлять его на наш новый эндпоинт.

**Результат:** Агент станет более гибким и сможет помогать с задачами, выходящими за рамки исходного проекта.
